# Wiki 内容质量审查报告 & 改进计划

> 以下是一位资深股票分析师对知识库 wiki 内容的全面审查。
> 审查日期：2026-04-19

---

## 一、总体评价：框架不错，内容很糙

知识库的**骨架是好的**——45 家公司、26 个行业、3 个主题的覆盖面足够广，graph.yaml 的产业链关系建模也合理。但**内容质量严重不达标**，大量页面是垃圾数据的堆砌，距离"可支撑投资决策的知识库"还有本质差距。

**一句话**：你建了一个很好的图书馆，但里面大部分书是印刷厂的废纸。

---

## 二、致命问题（必须立即修复）

### 问题 1：数据采集是垃圾进，wiki 就是垃圾出

**现状**：新闻采集脚本 (`collect_news.py`) 抓取了大量**无价值**的内容来源：
- 搜狐证券股票页面（只有价格数字）
- 东方财富股吧帖子（散户情绪宣泄）
- Yahoo Finance 页面导航文本
- 百度一下首页（就四个字"百度一下"）
- 腾讯证券空页面
- 公司官网产品目录页（产品型号列表）
- Moomoo/富途牛牛的新闻标题列表

**案例**：中际旭创（全球光模块龙头，市值数千亿）的 wiki 页面里：
```
### 2026-04-13 | 新闻 | 中际旭创股份有限公司-集光通信器件设计研发制造...
- 中际旭创股份有限公司-集光通信器件设计研发制造、智能装备制造于一身的技术创新型企业
```
这就是公司官网的 meta description，**一个字的分析都没有**。

```
### 2026-04-14 | 新闻 | 中際旭創 (300308.SZ) 股價、新聞、報價和記錄 - Yahoo 財經
- Skip to navigation Skip to right column. # 雅虎香港財經...
```
这是网页导航栏的文本，**完全无意义**。

**更严重的案例**：中际旭创页面出现了大量**乱码**（似乎是 GBK 编码的中文被当作其他编码读取）：
```
### 2026-04-15 | 新闻 | м(300308)ҵԤ_˲ƾ_
- | ҵԤ | Ԥ20251-3¹...
```
这意味着数据采集或存储环节存在**编码错误**。

**改进方案**：
1. 在 `collect_news.py` 中增加**内容质量过滤器**：
   - 过滤掉内容长度 < 200 字的页面
   - 过滤掉匹配已知垃圾模式的 URL（股吧、Yahoo Finance 导航、证券数据页）
   - 过滤掉网页导航文本（用 BeautifulSoup 提取正文，而非全页文本）
2. 在 `ingest.py` 中增加**去噪步骤**：如果原始新闻的正文信息密度太低（可以用 TF-IDF 或简单关键词密度判断），跳过不 ingest
3. 修复编码问题：统一使用 UTF-8，对抓取的内容做编码检测

### 问题 2：核心问题是模板化的废话

**现状**：很多公司的"核心问题"是完全相同的模板：
```
- XXX的核心竞争优势和护城河是什么？
- XXX的主要增长驱动力在哪里？
- XXX面临的最大风险和挑战是什么？
```

这三个问题放到**任何一家公司**上都适用，等于**什么都没问**。

**对比好的做法**：
- 台积电的核心问题写得很好——具体到"2nm 及以下产能利用率"、"GAAFET/CFET 商业化进展"、"AI 相关芯片营收占比季度变化"、"全球产能布局投产进度"
- 宁德时代也不错——具体到"凝聚态/麒麟/钠离子电池客户验证及量产时间表"、"储能业务收入占比及毛利率变动"

**改进方案**：
1. 参照 `graph.yaml` 中每个公司的 `position` 和 `sectors`，为核心问题设定**行业标准问题模板**。例如：
   - 光模块公司必须问：800G/1.6T 出货量、CPO 进展、海外大客户认证进度
   - 半导体设备公司必须问：各环节国产化率、先进制程验证进度、大客户订单
   - 代工公司必须问：产能利用率、ASP 趋势、先进制程占比
2. 重新生成所有公司的核心问题，确保每家公司至少有 3 个**行业特异化**的问题

### 问题 3：时间线条目是新闻搬运，不是分析

**现状**：大部分时间线条目只是**复制了原始新闻的标题和第一段**，没有任何分析判断。

**坏的案例**（中密控股）：
```
### 2026-04-14 | 新闻 | 中密控股- 密封- 零部件- 泵品牌- 工业泵网
- 中密控股拥有优泰科（苏州）密封技术有限公司、大连华阳密封股份有限公司...
```
这是工业泵网的公司简介页，复制过来**没有任何增量信息**。

**好的案例**（台积电）：
```
### 2026-04-17 | 研报 | TSMC reports 58% jump in profit...
- Taiwan's chipmaker TSMC has reported a 58% jump in profit for Q1
  thanks to strong demand driven by AI
```
这条至少有具体数据（58% profit jump），但仍然缺少"**这意味着什么**"的分析。

**改进方案**：
1. 在 ingest 流程中要求每个时间线条目必须包含：
   - 事实（发生了什么）
   - 数字（具体数据）
   - 判断（这对投资意味着什么）
2. 对于纯标题式条目（信息密度 < 阈值），要么不收录，要么标记为"待分析"

### 问题 4：行业页面大量无关内容

**现状**：EDA与IP 行业页面里出现了珂玛科技（半导体材料）、科大讯飞（AI应用）、紫光股份（算力基建）、中科曙光（算力基建）、新易盛（光模块）的内容。这些公司和 EDA/IP **毫无关系**。

```
### 2026-04-17 | 新闻 | 半导体设备用先进陶瓷材料零部件...
### 2026-04-12 | 产品 | 最新动态_科大讯飞介绍 - 亿欧
### 2026-04-18 | 财报 | 紫光股份：2020年年度报告摘要
```

这说明 ingest 脚本的**行业匹配逻辑**有严重问题——可能是按"包含关键词"匹配的，导致大量误分类。

**改进方案**：
1. ingest 时必须以 `graph.yaml` 中 `companies.XXX.sectors` 字段为**唯一权威来源**判断公司属于哪个行业
2. 关键词匹配只用于"相关动态"页面的补充推荐，不应作为主要分类依据

## 三、严重问题（应尽快修复）

### 问题 5：综合评估缺失或质量差

大部分公司页面缺少**综合评估**（`## 综合评估` 部分）。能找到的那些，质量参差不齐：
- 百度的评估有观点："AI搜索的彻底改造是决定其未来增长的关键一役"
- 但很多公司的评估要么缺失，要么是一般性陈述

**改进**：要求每个公司页面必须有综合评估，且必须包含：
1. 当前投资逻辑（1-2 句话）
2. 核心催化剂
3. 主要风险
4. 估值视角（不算目标价，但应讨论估值方法）

### 问题 6：竞争对手关系严重缺失

`graph.yaml` 中 45 家公司只有 **1 家**（AMD）定义了 `competes_with` 关系。这意味着：
- 无法回答"中微公司 vs 北方华创"的竞争格局
- 无法回答"中际旭创 vs 新易盛"的份额变化
- 图谱中缺少竞争维度的重要边

**改进**：为至少以下核心竞争对手组添加 `competes_with` 关系：
| 公司 | 竞争对手 |
|------|---------|
| 中际旭创 | 新易盛, 光迅科技 |
| 北方华创 | 中微公司 (刻蚀), 拓荆科技 (薄膜) |
| 台积电 | 中芯国际, 三星 |
| 英伟达 | AMD, 寒武纪, 海光信息 |
| 百度 | 科大讯飞, 商汤科技 |
| 宁德时代 | 比亚迪 (未收录) |
| 长电科技 | 通富微电 |

### 问题 7：行业页面内容同质化严重

很多行业页面本质上就是**所有相关公司的时间线条目大杂烩**，缺少行业层面的分析：
- 没有市场规模数据
- 没有增速预期
- 没有竞争格局（CR3/CR5）
- 没有技术路线图
- 没有政策影响分析

**改进**：每个行业页面应增加结构化章节：
```markdown
## 行业概况
- 市场规模（全球/中国）
- 增速预期（CAGR）
- 竞争格局（CR3/CR5，国产化率）

## 技术路线
- 主流技术路线
- 新技术方向
- 技术迭代节奏

## 政策环境
- 核心政策及影响
- 制裁/出口管制影响

## 投资要点
- 周期位置判断
- 核心跟踪指标
```

## 四、一般问题（持续改进）

### 问题 8：数据时效性标注不清

所有页面的 `last_updated` 都是 2026-04-18，但这不意味着所有内容都是新的。一些行业页面混入了 2015-2020 年的历史业绩预告（如中际旭创页面有 2015 年数据），这些应该被标记为"历史参考"或归档。

### 问题 9：缺少公司概览页

当前只有"公司动态"和"相关动态"两个页面。缺少一个**公司概览**页（type: overview），应包含：
- 主营业务及收入结构
- 股权结构（实际控制人）
- 核心财务指标（营收/净利/毛利率趋势）
- 在产业链中的定位

### 问题 10：tags 字段利用率低

大部分行业页面的 tags 是空数组 `[]`。公司页面的 tags 也过于简单（只有行业名）。应该利用 tags 做更细粒度的分类，例如：`[半导体设备, 刻蚀设备, ICP, 先进制程, 国产替代, 688012]`

## 六、改进实施计划

### Phase 1: 修复数据源（最高优先级）

**目标**：停止垃圾数据进入 wiki

1. **修改 `collect_news.py`**：
   - 增加 URL 黑名单（股吧、Yahoo Finance 导航页、证券数据页、公司官网产品页）
   - 增加正文提取（用 readability 或 trafilatura 提取网页正文，而非全页文本）
   - 增加内容长度过滤（正文 < 200 字的跳过）
   - 修复 GBK 编码问题
   - 添加来源质量评分（主流财经媒体 > 券商研报 > 公告 > 社区）

2. **修改 `ingest.py`**：
   - 增加原始新闻质量评估（信息密度检测）
   - 以 `graph.yaml` 的 sectors 字段为准进行行业匹配，不用关键词
   - 对低质量条目标记为"待验证"，不直接写入 wiki

### Phase 2: 重建核心问题（高优先级）

**目标**：每家公司有 3-5 个行业特异化的投资问题

1. 为每个 sector 定义 3-5 个标准问题
2. 为每家公司基于 position + sectors 定制问题
3. 批量重新生成所有公司的核心问题部分

### Phase 3: 清洗存量 wiki（中优先级）

**目标**：清除垃圾条目，提升 A/B 档比例

1. 编写清洗脚本，删除以下类型的条目：
   - 标题 = 正文（无增量信息）
   - 正文全是网页导航文本
   - 正文全是乱码
   - 正文 < 50 字
2. 对清洗后的页面重新计算 `sources_count`
3. 为缺失综合评估的页面生成评估

### Phase 4: 增强行业页面（中优先级）

**目标**：每个行业页面有结构化的行业分析

1. 为 26 个行业页面添加"行业概况"结构
2. 从研报中提取市场规模、竞争格局等数据
3. 添加竞争关系（`competes_with`）到 graph.yaml

### Phase 5: 增加公司概览页（低优先级）

1. 为每个公司创建 overview 类型的 wiki 页面
2. 包含主营业务、核心财务、产业链定位
3. 作为公司 wiki 目录的首个页面

## 八、验证标准

Phase 1 完成后：
- 新采集的新闻中，垃圾来源占比 < 5%
- 无编码错误
- 正文平均长度 > 500 字

Phase 2 完成后：
- 100% 公司有 ≥3 个行业特异化问题
- 0 公司使用通用模板问题

Phase 3 完成后：
- A/B 档占比 > 70%
- 每个时间线条目至少包含 1 个具体数据点
- 0 个纯标题条目
