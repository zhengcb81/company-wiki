## 二、设计审查

### 2.1 优点

**概念清晰**：
- 单一数据源设计（graph.yaml）
- 三层架构（raw → wiki → schema）
- 自动化流程设计

**模块化尝试**：
- 划分了 ingest、storage、monitoring 等模块
- 尝试使用 Repository 模式

### 2.2 严重问题

**问题1：设计与实现严重偏离**

原始设计理念：
```
Raw Sources → Wiki → Schema
     ↓
  LLM 自动整理
```

实际实现：
```
Raw Sources → 手动分类 → 错误目录 → 手动同步 → Wiki
```

**问题2：配置混乱**
- config.yaml、config.json、config_rules.yaml 三个配置文件
- 配置不一致导致文件下载到错误位置
- 缺乏配置验证机制

**问题3：缺乏清晰的 API 边界**
- 51个 Python 文件，59个重复函数名
- 函数命名混乱（load_config 出现6次）

### 3.2 架构问题

**问题1：循环依赖风险**
- graph.py 被导入 21 次
- 模块间耦合严重
- 缺乏清晰的依赖层次

**问题2：职责混乱**
- download_reports.py 包含下载、配置、文件组织逻辑
- ingest.py 包含扫描、提取、更新逻辑
- 单个文件超过 700 行

**问题3：缺乏抽象层**
- 直接操作文件系统
- 没有服务层或领域模型
- 业务逻辑与基础设施混杂

## 四、实现审查

### 4.1 代码质量统计

| 指标 | 数值 | 评价 |
|------|------|------|
| 总文件数 | 51 | 过多 |
| 总代码行数 | 14,896 | 中等 |
| 平均行数/文件 | 292 | 合理 |
| 最大文件 | 718 行 (query.py) | 过大 |
| 重复函数名 | 59 个 | 严重 |

### 4.2 代码质量问题

**问题1：错误处理不一致**
```python
# 方式1: sys.exit
print("ERROR: xxx")
sys.exit(1)

# 方式2: 抛出异常
raise ValueError("xxx")

# 方式3: 返回空值
return []

# 方式4: 静默失败
except Exception:
    pass
```

**问题2：过度使用 print**
- 46 个文件使用 print 输出
- 应该使用 logging 模块
- 缺乏统一的日志格式

**问题3：硬编码路径**
```python
# download_reports_v2.py
WINDOWS_DOWNLOADS = Path("/mnt/c/Users/郑曾波/Projects/StockInfoDownloader/downloads")

# sync_from_windows.py
DOWNLOADER_DIR = Path.home() / "StockInfoDownloader"
```

**问题4：配置验证缺失**
- 下载前不验证配置
- 下载后不验证结果
- 配置错误导致静默失败

**问题5：缺乏事务性**
- 文件复制没有原子性
- 中断可能导致不一致状态
- 没有回滚机制

### 4.3 具体问题示例

**示例1：download_reports.py 的致命缺陷**
```python
def run_download(stock_code, company_name, page_suffix):
    # 1. 修改 config.json
    config["save_dir"] = str(WIKI_COMPANIES)
    
    # 2. 调用下载
    result = subprocess.run(...)
    
    # 3. 检查文件 - 但文件在错误位置！
    wiki_company_dir = WIKI_COMPANIES / company_name / "raw"
    if wiki_company_dir.exists():
        files = list(wiki_company_dir.rglob("*.pdf"))
        if files:
            return files  # 永远不会执行到这里
    
    return []  # 总是返回空列表
```

## 七、核心问题总结

### 7.1 致命问题

1. **基本功能不可靠**：下载文件到指定目录这种基本功能都做不好
2. **配置混乱**：三个配置文件，配置不一致
3. **缺乏验证**：没有验证配置是否生效，文件是否正确
4. **测试缺失**：41个模块没有测试

### 7.2 严重问题

1. **代码质量差**：59个重复函数名，错误处理不一致
2. **架构混乱**：模块耦合严重，职责不清
3. **文档缺失**：没有 README，API 文档不完整
4. **硬编码**：路径硬编码，配置硬编码

### 7.3 一般问题

1. **过度使用 print**：应该用 logging
2. **代码重复**：append_log 等函数重复
3. **缺乏事务性**：文件操作没有原子性
